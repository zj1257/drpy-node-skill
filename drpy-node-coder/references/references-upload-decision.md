# 仓库上传判断参考（reference）

> 供 `drpy-node-repo-upload` 在做上传建议时使用。
> 目标不是判断“技术上能不能上传”，而是判断“现在是否建议上传当前版本”。

---

## 一、三档判断模型

### A 档：建议上传
满足示例：
- 语法通过
- 结构通过
- 关键接口已通
- 用户当前目标就是发布/替换当前版本

### B 档：技术上可传，但不建议直接上传
满足示例：
- 语法和结构通过
- 但边界问题仍明显
- fallback 搜索被误当成完整搜索
- 播放只在单测中可用，但用户要求的是“修好最终版”
- 用户表达的是“修好后再传”而不是“先传再说”

### C 档：暂不应上传
满足示例：
- 语法报错
- 结构无效
- detail 为空
- play 仍是假通过

---

## 二、三级验证深度标准

A/B/C 档判断不应仅基于 `drpy_check_syntax` + `validate_spider`。按验证深度分层：

| 级别 | 覆盖范围 | 能支撑的判断 |
|------|---------|-------------|
| L1: 语法结构 | `drpy_check_syntax` + `validate_spider` | 仅说明“非残档”，可进入下一步，不能判断是否可上传 |
| L2: 单接口 | `test_spider_interface(home/category/detail/play)` | 判断具体接口通断；可支撑窄范围技术上传 |
| L3: 全流程 | `evaluate_spider_source` | 综合评分，支持 A/B/C 发布建议 |

**禁止**在没有 L2/L3 验证时就给出“A 档建议上传”的结论。“最终版/修好后上传”必须有 L3；L2 只适合说明某个已验证接口或窄范围修复可用。

### L3 分数模型

`evaluate_spider_source` 总分 100：home 20、category 20、detail 25、play 25、search 10。一级 + 二级 + 播放是有效性核心；search 丢分可以降档说明，但不能掩盖 detail/play 断链。

### 自主最终版上传标准

当用户原始目标是“自动完成 / 修到100 / 满分后上传”时，上传门槛收紧为：

- 必须有 L3 证据，且 `evaluate_spider_source == 100`。
- 必须是 A 档，不能用 B 档替代最终版。
- 必须明确本地路径、tags、is_public、auto_replace 策略和目标对象。
- L3<100 时返回 `score_below_target`，交回 workflow/source-create 继续修复或报告 blocker。

只有用户明确放宽目标（例如“先传再说”）时，才允许在 B 档风险说明后上传。

## 三、自动标签检测机制

`house_file` 的 `upload` 操作内部有自动类型检测逻辑：

| 检测条件 | 自动附加的标签 |
|---------|--------------|
| `spider/js/` 下的 `.js` 文件 | `ds` |
| `spider/catvod/` 下的文件 | `catvod` |
| 文件名含 `[官]`/`[优]` | `优` |
| 内容含 `jx:`/`parse:` | `jx` |
| 内容含 `hipy`/`海阔视界` | `hipy` |
| `.php` 文件或含 `<?php` | `php` |

用户没有明确要求标签时，可利用自动检测作为默认值。
但用户明确要求标签时，以用户要求为准。

## 四、metadata 与仓库 tags

源内 metadata 与仓库 tags 是两套概念：
- `@header` / rule metadata：运行时身份和能力，如 title、类型、lang、searchable、filterable、quickSearch。
- 仓库 tags：上传记录的检索标签，由用户要求或 `house_file(upload)` 自动检测决定。

用户明确要求 tags 时必须优先用户要求；自动检测只作为默认。上传后用 `house_file(info)` 核验 tags，不一致时再询问是否修正。

---

## 五、特殊内容上传标准

特殊内容源的上传判断不能强套 m3u8/mp4：

| 类型 | 上传前重点 |
|---|---|
| 漫画 | `@header` 类型为漫画，detail 有章节，play/lazy 返回 `pics://` 图片列表 |
| 小说 | `@header` 类型为小说，detail 有章节，play/lazy 返回 `novel://` 或正文内容 |
| 音乐/听书 | 类型与内容一致，play/lazy 返回 mp3/m4a 等音频直链 |
| 网盘/投屏 | 输出 `push://` 或网盘专用可用结果，不伪装成普通视频直链 |

## 六、判断时必须关注的边界问题

### 1. fallback 搜索不是完整搜索
如果一个源当前采用的是：
- suggest 搜索
- RSS 搜索

则要明确：
- 通常不支持翻页
- 不能自动等价视为完整原生搜索能力

### 2. 单接口通过不等于整体发布无风险
例如：
- 单独 play 通
- 但 detail 仍不稳定
- 或 evaluate 串联没有完全站住

这种情况下，可能属于 B 档，而不是 A 档。

### 3. 用户目标影响上传建议
同一份源：
- 用户说“先传我再慢慢修” → 可偏向 B 档执行上传
- 用户说“修好最终版再传” → B 档通常应建议先别传

---

## 七、推荐汇报方式

上传前应明确汇报：
- 当前属于 A / B / C 哪一档
- 依据是什么
- 如果仍要上传，风险点是什么

---

## 八、核心原则

1. 技术上能上传 ≠ 现在建议上传
2. 用户目标决定建议强度
3. fallback 搜索 / 边界问题必须说清楚
4. 上传后结果必须可追踪（file_id / cid / tags）
